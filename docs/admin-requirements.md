# LaneBlog 项目需求文档

## 1. 文档说明

### 1.1 项目名称

LaneBlog

### 1.2 本阶段目标

本阶段先产出后台管理系统需求，服务于 LaneBlog 的内容管理与基础运营。系统分为前台和后台两部分：

- 前台：博客访客浏览文章、分类、标签等内容。
- 后台：管理员维护文章分类、文章内容、标签数据及登录态。

本需求文档重点定义后台管理系统，后端语言为 Go，数据库为 MySQL，给定表结构视为既有事实，不在本阶段调整表结构。

### 1.3 设计原则

- 以现有表结构为核心进行功能设计。
- 后台仅支持单管理员登录，不设计多角色权限系统。
- 在不改表结构的前提下，尽量保证数据一致性与操作可用性。
- 优先满足文章发布、分类维护、标签联动这三类核心场景。

## 2. 项目范围

### 2.1 包含范围

后台管理系统需至少包含以下模块：

1. 登录与会话管理
2. 后台首页/控制台
3. 文章分类管理
4. 文章管理
5. 标签管理
6. 基础系统能力（搜索、分页、状态提示、操作日志建议项）

### 2.2 不包含范围

以下内容不作为本阶段必须交付项：

1. 前台页面开发
2. 用户注册、找回密码、多管理员权限
3. 评论系统
4. 文件上传系统
5. SEO 静态化生成
6. 推荐算法、消息通知、草稿协同编辑

## 3. 用户角色

本系统仅有一个后台角色：

- 管理员：负责登录后台，维护分类、文章、标签。

## 4. 登录需求

### 4.1 登录方式

后台采用固定账号密码登录：

- 用户名：`lane`
- 密码校验规则：输入密码后，先进行 MD5，再将 MD5 结果字符串进行 SHA1，最终与后端代码中的固定值比对
- 固定校验值：`58cd2c961705ac711493b038c04bb98c7c926ed0`

即：

```text
sha1(md5("******")) = 58cd2c961705ac711493b038c04bb98c7c926ed0
```

### 4.2 登录页面需求

页面元素包含：

1. 用户名输入框
2. 密码输入框
3. 登录按钮
4. 错误提示信息区

交互要求：

1. 用户名或密码为空时，前端阻止提交并提示必填。
2. 用户名错误时，提示“用户名或密码错误”。
3. 密码错误时，提示“用户名或密码错误”。
4. 登录成功后跳转到后台首页。
5. 登录成功后写入服务端会话或签发后台访问令牌。

### 4.3 安全要求

1. 后台所有业务接口均需校验登录态。
2. 未登录访问后台页面时，统一跳转登录页。
3. 登出后会话立即失效。
4. 固定账号密码写在服务端配置或代码常量中，不下发到前端。
5. 尽管本项目为个人博客后台，仍建议对登录接口增加基础限流能力。

## 5. 信息架构

后台左侧导航建议如下：

1. 控制台
2. 分类管理
3. 文章管理

5. 退出登录

## 6. 数据表映射说明

### 6.0 数据库字符集要求

LaneBlog 后台使用的 MySQL 数据库及相关表统一要求如下：

- 字符集：`utf8mb4`
- 排序规则：`utf8mb4_unicode_ci`

涉及的核心表包括：

1. `info_article`
2. `info_menu`
3. `info_tag`

若现有库表仍为旧字符集，需统一调整为以下形式：

```sql
ALTER TABLE info_article CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
ALTER TABLE info_menu CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
ALTER TABLE info_tag CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

### 6.1 文章表 `info_article`

用于存储博客文章主体信息，主要字段含义如下：

- `id`：文章 ID
- `mid`：所属分类 ID，对应 `info_menu.id`
- `author`：作者
- `title`：文章标题
- `description`：文章摘要
- `seo_title`：SEO 标题
- `seo_description`：SEO 描述
- `seo_keywords`：SEO 关键词
- `tag`：文章标签，使用字符串存储
- `clicks`：点击次数
- `content`：文章正文
- `ctime`：创建时间，Unix 时间戳
- `good_num`：点赞次数
- `bad_num`：点踩次数
- `recommend_type`：推荐类型，`1` 为全站推荐，`2` 为首页推荐

### 6.2 分类表 `info_menu`

用于存储文章分类，主要字段含义如下：

- `id`：分类 ID
- `name`：分类名称
- `seo_title`：SEO 标题
- `seo_description`：SEO 描述
- `seo_keywords`：SEO 关键词
- `in_out`：链接类型，`1` 为站内，`2` 为站外
- `pid`：父分类 ID，`0` 表示顶级分类
- `url`：站外链接地址，当 `in_out=2` 时生效
- `item`：分类标识字段，保留按原表使用

### 6.3 标签表 `info_tag`

用于存储标签聚合数据：

- `id`：标签 ID
- `tag`：标签内容
- `num`：标签出现次数

## 7. 后台首页需求

### 7.1 页面目标

后台首页用于展示博客内容管理概览，帮助管理员快速了解当前数据规模。

### 7.2 展示内容

建议展示以下统计卡片：

1. 文章总数
2. 分类总数
3. 标签总数
4. 首页推荐文章数
5. 全站推荐文章数

建议展示以下快捷区域：

1. 最近发布文章列表
2. 快捷入口：新建文章
3. 快捷入口：新建分类

## 8. 分类管理需求

### 8.1 功能目标

管理员可对文章分类进行新增、编辑、列表查看和删除操作，支持顶级分类和子分类。

### 8.2 分类列表

列表字段建议如下：

1. 分类 ID
2. 分类名称
3. 父级分类
4. 类型（站内/站外）
5. 链接地址
6. SEO 标题
7. 分类标识 `item`
8. 操作项（编辑、删除）

列表能力要求：

1. 支持按分类名称搜索。
2. 支持树形或缩进形式展示父子关系。
3. 默认按 `id` 升序展示。

### 8.3 新增分类

表单字段：

1. 分类名称 `name`
2. SEO 标题 `seo_title`
3. SEO 描述 `seo_description`
4. SEO 关键词 `seo_keywords`
5. 链接类型 `in_out`
6. 父级分类 `pid`
7. 出站链接 `url`
8. 分类标识 `item`

校验规则：

1. `name` 必填，长度不超过 50。
2. `seo_title` 必填，长度不超过 100。
3. `seo_description` 必填，长度不超过 500。
4. `seo_keywords` 必填，长度不超过 200。
5. `in_out` 仅允许 `1` 或 `2`。
6. `pid` 必须为 `0` 或已存在的分类 ID。
7. 当 `in_out=2` 时，`url` 必填且必须为合法链接。
8. 当 `in_out=1` 时，`url` 可为空或保存为空字符串。
9. `item` 必填，长度不超过 20。

业务规则：

1. 支持创建顶级分类和二级分类。
2. 本阶段默认不支持超过二级的深层分类。
3. 分类名称允许重复，但不建议；若实现方便，建议同级分类名唯一。

### 8.4 编辑分类

编辑规则与新增一致，同时要求：

1. 编辑后立即影响文章分类归属展示。
2. 不允许将分类父级设置为自身。
3. 不允许形成循环父子关系。

### 8.5 删除分类

删除规则：

1. 若分类下存在子分类，不允许直接删除。
2. 若分类下存在文章，不允许直接删除。
3. 删除失败时需明确提示原因。

## 9. 文章管理需求

### 9.1 功能目标

管理员可对文章进行新增、编辑、列表查看、搜索、删除和推荐设置。

### 9.2 文章列表

列表字段建议如下：

1. 文章 ID
2. 标题
3. 分类名称
4. 作者
5. 标签
6. 点击数
7. 点赞数
8. 点踩数
9. 推荐类型
10. 创建时间
11. 操作项（编辑、删除）

列表能力要求：

1. 支持按标题关键字搜索。
2. 支持按分类筛选。
3. 支持按推荐类型筛选。
4. 支持分页。
5. 默认按创建时间倒序展示。

### 9.3 新增文章

表单字段：

1. 所属分类 `mid`
2. 作者 `author`
3. 标题 `title`
4. 摘要 `description`
5. SEO 标题 `seo_title`
6. SEO 描述 `seo_description`
7. SEO 关键词 `seo_keywords`
8. 标签 `tag`
9. 初始点击数 `clicks`
10. 正文内容 `content`
11. 创建时间 `ctime`
12. 点赞次数 `good_num`
13. 点踩次数 `bad_num`
14. 推荐类型 `recommend_type`

校验规则：

1. `mid` 必填，且必须是有效分类 ID。
2. `author` 必填，长度不超过 50。
3. `title` 必填，长度不超过 100。
4. `description` 必填，长度不超过 500。
5. `seo_title` 必填，长度不超过 100。
6. `seo_description` 必填，长度不超过 500。
7. `seo_keywords` 必填，长度不超过 200。
8. `tag` 必填。
9. `content` 必填。
10. `clicks`、`good_num`、`bad_num` 必须为大于等于 0 的整数。
11. `recommend_type` 仅允许 `1` 或 `2`；若业务希望支持“不推荐”，可在服务层约定 `0`，但当前表注释仅声明 `1/2`，本阶段以 `1/2` 为准。
12. `ctime` 默认写入当前时间戳，也允许管理员手动指定发布时间。

标签录入规则：

1. 标签字段使用英文逗号分隔，例如：`Go,MySQL,后台管理`。
2. 前后空格在保存时自动裁剪。
3. 空标签项自动忽略。
4. 单篇文章建议最多 10 个标签。

编辑器要求：

1. 正文编辑器至少支持纯文本和基础富文本输入。
2. 若后续前台以 HTML 渲染，则后台需支持 HTML 内容保存。
3. 若本阶段不接入图片上传，编辑器中的图片可先仅支持外链。

### 9.4 编辑文章

编辑规则与新增一致，并补充：

1. 编辑文章后需同步更新标签统计。
2. 若文章分类被修改，列表和前台归属需同步更新。
3. 点击数、点赞数、点踩数允许后台人工修正。

### 9.5 删除文章

删除规则：

1. 支持单篇删除。
2. 删除文章后需同步更新标签统计。
3. 删除成功后列表刷新并提示成功。

### 9.6 推荐逻辑

基于现有字段 `recommend_type`，定义如下：

1. `1`：全站推荐
2. `2`：首页推荐

说明：

1. 当前字段设计仅支持单值，不支持一篇文章同时属于两种推荐。
2. 若后续需要同时多选，需调整表结构或新增推荐关系表。

## 10. 标签管理需求

### 10.1 功能目标

后台需提供标签查看和维护能力，标签数据主要由文章保存逻辑自动维护。

### 10.2 标签列表

列表字段建议如下：

1. 标签 ID
2. 标签名称
3. 文章引用次数
4. 操作项

列表能力要求：

1. 支持按标签名称搜索。
2. 支持按引用次数排序。

### 10.3 标签维护规则

1. 新增或编辑文章时，系统自动解析 `info_article.tag` 字段。
2. 若标签不存在，则自动写入 `info_tag`。
3. 若标签已存在，则对应 `num` 加 1 或按实际文章数量重算。
4. 删除文章或修改文章标签后，需同步修正 `info_tag.num`。
5. 当某标签引用次数为 0 时，可保留记录，也可自动删除；建议自动删除以避免脏数据。

### 10.4 标签后台操作

本阶段建议提供以下能力：

1. 查看标签列表
2. 删除标签

删除标签规则：

1. 删除标签时，应同步从相关文章的 `tag` 字段中移除，或禁止直接删除。
2. 为避免字符串同步修改复杂度过高，本阶段更建议“禁止删除被文章引用的标签”。

## 13. 后端业务规则

### 13.1 分类与文章关系

1. 文章所属分类必须存在。
2. 站外分类原则上不应被文章引用；若 `in_out=2` 表示纯外链栏目，则建议后台在文章归类时过滤掉该类分类。
3. 若业务允许文章挂到站外分类，需在前台跳转逻辑中做额外兼容，但本阶段不建议。

### 13.2 标签统计策略

为降低字符串标签带来的统计误差，建议采用“重算优先”的策略：

1. 新增文章后，解析标签并同步标签表。
2. 编辑文章时，对比旧标签和新标签，更新标签表。
3. 删除文章时，更新标签表。
4. 提供后台内部标签重建函数，必要时可全量重算 `info_tag`。

### 13.3 时间处理

1. `ctime` 使用 Unix 时间戳存储。
2. 后台展示时统一格式化为本地时间。

## 14. 非功能需求

### 14.1 技术要求

1. 后端使用 Go。
2. 数据库使用 MySQL。
3. 后台前端技术栈可独立选型，但需与 Go 后端接口对接。

### 14.2 性能要求

1. 后台列表页在正常数据量下应能快速返回。
2. 列表接口需支持分页，避免全量加载。
3. 分类和标签数据量较小时可直接查询，文章列表必须分页。

### 14.3 可靠性要求

1. 所有新增、编辑、删除操作需有错误处理。
2. 删除、保存失败时不可造成明显脏数据。
3. 涉及文章与标签联动时，建议使用事务或补偿逻辑；若 MyISAM 不支持事务，则需在服务层谨慎控制写入顺序与失败回滚策略。

## 15. 验收标准

### 15.3 文章管理验收

1. 可新增文章并正确入库。
2. 可编辑文章并正确保存。
3. 可删除文章。
4. 文章列表支持搜索、筛选、分页。
5. 文章与分类关系正确。
6. 推荐类型保存和展示正确。

### 15.4 标签管理验收

## 19. 后台原型

### 19.1 原型目标

后台原型用于明确 LaneBlog 管理后台的页面结构、信息布局、核心交互与跳转关系，供 UI 设计、前端实现和后端接口联调共同使用。

### 19.2 整体布局原型

后台整体采用经典管理系统布局：

1. 顶部区域：展示系统名称 `LaneBlog Admin`、当前登录账号、退出登录入口。
2. 左侧导航：展示后台模块菜单。
3. 右侧主内容区：展示列表页、表单页、统计页等业务内容。
4. 面包屑区域：展示当前页面位置，如“控制台 / 文章管理 / 新增文章”。

原型结构示意：

```text
+--------------------------------------------------------------+
| LaneBlog Admin                            lane | 退出登录     |
+----------------------+---------------------------------------+
| 控制台               | 面包屑：控制台                        |
| 分类管理             |---------------------------------------|
| 文章管理             |                                       |
| 标签管理             |            主内容区域                 |
|                      |                                       |
|                      |                                       |
+----------------------+---------------------------------------+
```

### 19.3 登录页原型

页面目标：

1. 提供唯一管理员登录入口。
2. 风格简洁，突出品牌名称和登录动作。

页面结构：

1. 页面标题：`LaneBlog Admin`
2. 登录卡片
3. 用户名输入框
4. 密码输入框
5. 登录按钮
6. 错误提示区域

原型示意：

```text
+----------------------------------+
|         LaneBlog Admin           |
|----------------------------------|
| 用户名 [______________________]  |
| 密码   [______________________]  |
|                                  |
| [            登录             ]  |
|                                  |
| 错误提示信息                     |
+----------------------------------+
```

交互说明：

1. 登录成功后进入控制台。
2. 登录失败后在卡片内提示错误信息。
3. 回车可触发登录。

### 19.4 控制台原型

页面目标：

1. 展示后台核心统计信息。
2. 提供常用快捷操作入口。

页面结构：

1. 数据概览卡片区
2. 最近文章区
3. 快捷入口区

原型示意：

```text
+--------------------------------------------------------------+
| 文章总数 | 分类总数 | 标签总数 | 全站推荐 | 首页推荐         |
|   120    |   12     |   85     |    8     |    5            |
+--------------------------------------------------------------+
| 最近发布文章                                                 |

| 3. 博客系统设计说明                                          |
+--------------------------------------------------------------+
| [新建文章]   [新建分类]   [查看标签]                         |
+--------------------------------------------------------------+
```

### 19.5 分类管理页原型

页面目标：

1. 查看分类层级关系。
2. 进行新增、编辑、删除分类操作。

页面结构：

1. 搜索栏
2. 新增分类按钮
3. 分类列表表格

原型示意：

```text
+--------------------------------------------------------------+
| 分类名称 [__________]  [搜索]                 [新增分类]      |
+--------------------------------------------------------------+

| 2  | Go       | 后端开发 | 站内 | -        | go   | 编辑 删除 |
```

## 20. 页面清单

### 20.1 页面总览

LaneBlog 后台一期建议包含以下页面：

1. 登录页
2. 控制台首页
3. 分类列表页
4. 新增分类页
5. 编辑分类页
6. 文章列表页
7. 新增文章页
8. 编辑文章页
9. 标签列表页

### 20.2 页面清单明细

#### 20.2.1 登录页

页面路径建议：`/admin/login`

页面功能：

1. 管理员账号登录
2. 登录失败提示
3. 登录成功跳转控制台

核心组件：

1. 用户名输入框
2. 密码输入框
3. 登录按钮
4. 错误提示区

#### 20.2.2 控制台首页

页面路径建议：`/admin`

页面功能：

1. 展示后台统计总览
2. 展示最近文章
3. 提供快捷入口

核心组件：

1. 统计卡片
2. 最近文章列表
3. 快捷操作按钮

#### 20.2.3 分类列表页

页面路径建议：`/admin/category`

页面功能：

1. 分类查询
2. 分类列表查看
3. 编辑和删除分类
4. 进入新增分类页面

核心组件：

1. 搜索框
2. 新增按钮
3. 分类表格
4. 分页器

#### 20.2.4 新增分类页

页面路径建议：`/admin/category/create`

页面功能：

1. 创建顶级分类或子分类
2. 录入 SEO 信息
3. 配置站内/站外属性

核心组件：

1. 分类表单
2. 保存按钮
3. 取消按钮

#### 20.2.5 编辑分类页

页面路径建议：`/admin/category/edit/:id`

页面功能：

1. 修改已有分类信息
2. 调整父级关系
3. 调整链接属性

核心组件：

1. 分类表单
2. 保存按钮
3. 删除入口

#### 20.2.6 文章列表页

页面路径建议：`/admin/article`

页面功能：

1. 文章搜索
2. 分类筛选
3. 推荐类型筛选
4. 文章编辑和删除
5. 进入新增文章页面

核心组件：

1. 搜索栏
2. 筛选下拉框
3. 新建文章按钮
4. 文章表格
5. 分页器

#### 20.2.7 新增文章页

页面路径建议：`/admin/article/create`

页面功能：

1. 创建文章
2. 录入 SEO 信息
3. 维护标签、推荐类型、正文内容

核心组件：

1. 文章表单
2. 富文本或文本编辑器
3. 保存按钮
4. 保存并返回按钮
5. 取消按钮

#### 20.2.8 编辑文章页

页面路径建议：`/admin/article/edit/:id`

页面功能：

1. 修改文章内容
2. 修改标签和推荐类型
3. 修改文章统计字段

核心组件：

1. 文章表单
2. 正文编辑器
3. 保存按钮
4. 删除入口

#### 20.2.9 标签列表页

页面路径建议：`/admin/tag`

页面功能：

1. 标签搜索
2. 标签列表查看
3. 删除未被引用的标签

核心组件：

1. 搜索框
2. 标签表格
3. 分页器

### 20.3 页面跳转关系

页面间跳转建议如下：

1. 登录页登录成功后跳转控制台首页。
2. 控制台首页可跳转到分类管理、文章管理、标签管理。
3. 分类列表页可跳转新增分类页和编辑分类页。
4. 文章列表页可跳转新增文章页和编辑文章页。
5. 标签列表页可通过左侧导航直接进入。

### 20.4 页面优先级

为便于开发排期，建议按以下优先级实现：

1. P0：登录页、控制台首页、分类列表页、文章列表页、文章编辑页
2. P1：新增分类页、编辑分类页、标签列表页
3. P2：控制台增强统计、列表高级筛选、表单体验优化
