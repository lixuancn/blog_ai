# LaneBlog 前台技术方案

## 1. 文档目标

本文档用于定义 LaneBlog 前台网站的技术实现方案，覆盖首页、文章详情页、导航结构、接口设计、数据查询规则和前后端职责分工，作为后续前台开发的依据。

## 2. 方案结论

### 2.1 推荐实现方式

前台建议采用：

- 后端：Go
- 渲染方式：SSR 服务端渲染
- 模板方案：Go 原生 `html/template`
- 样式方案：原生 CSS 或轻量 CSS 体系
- 前端交互：少量原生 JavaScript

### 2.2 为什么采用 SSR

个人博客前台以内容展示和 SEO 为主，SSR 更适合当前项目：

1. 首页和文章详情页天然适合服务端输出 HTML。
2. 搜索引擎更容易抓取完整内容。
3. 技术复杂度低，不需要单独维护一套前台 SPA 工程。
4. 能与现有 Go 后端直接整合，部署简单。
5. 当前前台交互较轻，没必要引入重型前端框架。

### 2.3 不推荐的方案

当前阶段不建议优先采用：

1. React/Vue 单页应用
2. 前后端完全分离的前台工程
3. 静态生成方案

原因是当前需求页面较少，交互简单，而 SEO 和内容首屏对博客更重要。

## 3. 页面范围

本阶段前台至少包含以下页面：

1. 首页
2. 文章详情页

为了支撑导航和层级关系，建议预留以下页面能力：

1. 分类页
2. 外链分类跳转

其中分类页可作为下一阶段页面，但路由和数据结构建议本阶段一并设计。

## 4. 信息架构

### 4.1 顶部导航

顶部导航基于 `info_menu` 表生成，要求支持层级关系。

规则如下：

1. `pid=0` 的记录作为顶级导航。
2. `pid>0` 的记录作为子导航。
3. `in_out=1` 表示站内导航，点击进入站内页面。
4. `in_out=2` 表示站外导航，点击后跳转到 `url`。

建议导航交互：

1. 顶级导航直接展示在顶部栏。
2. 有子级的顶级导航，鼠标悬停或点击后显示下拉菜单。
3. 移动端可折叠为汉堡菜单。

### 4.2 站点头部

导航栏下方展示站点标题区：

- 主标题：`LaneBlog`
- 副标题：`每一个没有起舞的日子都是在辜负生命。`

建议该区域在首页突出展示，在文章详情页保留较紧凑版本。

## 5. 首页方案

### 5.1 页面结构

首页分为三部分：

1. 顶部导航
2. 站点标题区
3. 主内容区

主内容区采用双栏布局：

1. 左侧文章列表主栏
2. 右侧推荐文章侧栏

### 5.2 首页展示规则

文章列表按 `ctime` 倒序排列。

首页文章列表需支持分页，建议规则如下：

1. 使用查询参数 `?page=` 控制页码。
2. 默认每页展示 10 篇文章。
3. 当页码无效时，自动回落到第 1 页。
4. 页面底部展示上一页、下一页和当前页信息。
5. 推荐侧栏不跟随分页裁剪，保持独立展示。

每篇文章展示：

1. 标题
2. 作者
3. 发布时间
4. 分类名
5. 标签
6. 摘要 `description`
7. 正文前 100 个字
8. “阅读全文”入口

### 5.3 正文截断规则

首页只显示正文前 100 个字，建议规则如下：

1. 若正文存储为 HTML，先去掉 HTML 标签再截取。
2. 按 Unicode 字符数截取，不按字节截取。
3. 超过 100 个字时追加 `...`。
4. 若 `description` 已完整表达摘要，则正文截断可作为补充信息。

建议优先展示顺序：

1. 标题
2. 摘要
3. 正文前 100 字

### 5.4 右侧推荐文章

首页右侧展示“全站推荐文章列表”。

规则如下：

1. 只取 `recommend_type=1` 的文章。
2. 按 `ctime` 倒序展示。
3. 只展示标题。
4. 点击后进入文章详情页。

建议数量：

1. 默认展示 8 到 10 条。

## 6. 文章详情页方案

### 6.1 页面结构

文章详情页分为：

1. 顶部导航
2. 文章正文主栏
3. 右侧推荐栏
4. 底部操作区

### 6.2 文章详情展示内容

文章详情页展示全部信息，建议包含：

1. 标题
2. 作者
3. 发布时间
4. 分类名
5. 标签
6. 正文完整内容
7. 点击数
8. 点赞数
9. 点踩数
10. SEO 信息用于页面 `title` 和 `meta`

### 6.3 点击计数规则

每次进入文章详情页，文章点击数 `clicks` 自动加 1。

建议处理规则：

1. 页面请求详情成功后，由后端完成点击数累加。
2. 先更新点击数，再返回文章详情，或先读取再异步加 1，均可。
3. 为保证简单可靠，当前阶段建议在后端详情逻辑中同步加 1。

### 6.4 点赞与点踩

页面底部提供两个按钮：

1. 赞一下
2. 踩一下

对应数据库字段：

1. `good_num`
2. `bad_num`

建议规则：

1. 点击“赞一下”后，`good_num + 1`
2. 点击“踩一下”后，`bad_num + 1`
3. 当前阶段不做登录限制
4. 当前阶段不做人机校验
5. 当前阶段不限制同一用户重复点击

说明：

这意味着当前统计是宽松型数据统计，适合个人博客早期版本。若后续需要防刷，再增加 IP、Cookie 或设备级限制。

### 6.5 右侧类目推荐

文章详情页右侧展示“该类目下的推荐文章列表”。

建议规则：

1. 优先取当前文章同分类 `mid` 的文章。
2. 过滤当前文章自身。
3. 优先展示 `recommend_type=2` 的文章。
4. 若当前分类下 `recommend_type=2` 数量不足，可补同类最新文章。
5. 只展示标题。

建议数量：

1. 默认展示 6 到 10 条。

## 7. 路由方案

### 7.1 页面路由

建议前台采用以下路由：

1. `/`：首页
2. `/article/:id`：文章详情页
3. `/category/:id`：分类页，预留

### 7.2 导航跳转规则

1. 站内顶级分类跳转到分类页或对应文章聚合页。
2. 站内子分类也跳转到分类页。
3. 站外导航直接跳到 `url`。

若本阶段暂不做分类页，可以先将站内导航统一跳到：

1. `/`
2. 或 `/category/:id` 占位页

但从技术完整性看，建议本阶段同时把分类页能力预留。

## 8. 数据查询方案

### 8.1 首页数据

首页需要以下数据：

1. 导航树
2. 文章列表
3. 全站推荐列表
4. 站点信息

建议查询逻辑：

1. 查询 `info_menu` 构建导航树
2. 查询 `info_article`，按 `ctime DESC` 获取文章列表
3. 查询 `recommend_type=1` 的文章作为全站推荐

### 8.2 文章详情数据

文章详情页需要以下数据：

1. 当前文章详情
2. 导航树
3. 当前分类下推荐文章

建议查询逻辑：

1. 按 `id` 查询文章详情
2. 将当前文章 `clicks + 1`
3. 按 `mid` 查询同分类推荐文章

## 9. 后端接口方案

当前前台建议由 Go 后端直接渲染页面，但仍建议在服务层拆分出明确的数据接口能力。

### 9.1 前台页面服务接口

建议至少提供以下服务方法：

1. 获取导航树
2. 获取首页文章列表
3. 获取首页推荐文章
4. 获取文章详情并累加点击数
5. 获取文章详情侧栏推荐
6. 文章点赞
7. 文章点踩

### 9.2 若保留 JSON API

可设计如下接口：

1. `GET /api/v1/front/home`
2. `GET /api/v1/front/articles/:id`
3. `POST /api/v1/front/articles/:id/good`
4. `POST /api/v1/front/articles/:id/bad`
5. `GET /api/v1/front/categories/:id/articles`

但如果采用纯 SSR，也可以不单独暴露这些前台 JSON API，而在 Go handler 内直接调用 service。

## 10. 服务端渲染方案

### 10.1 模板拆分建议

建议模板拆分为：

1. `layout`
2. `header`
3. `footer`
4. `home`
5. `article_detail`
6. `sidebar_recommend`
7. `nav_tree`

### 10.2 页面渲染职责

后端 handler 负责：

1. 取数
2. 组装页面 ViewModel
3. 输出 HTML
4. 注入 SEO 信息

前端脚本负责：

1. 点赞请求
2. 点踩请求
3. 导航栏移动端交互

## 11. 前端样式方案

### 11.1 设计方向

建议前台风格偏简洁、阅读优先，突出博客内容。

建议布局：

1. 页面宽度固定在 1100 到 1280 像素之间
2. 正文区宽于侧栏
3. 标题字号明显高于摘要和正文信息
4. 保持足够留白

### 11.2 响应式要求

建议至少支持：

1. PC 端双栏布局
2. 平板端窄双栏或单栏
3. 手机端单栏布局

移动端策略：

1. 右侧推荐栏下移到底部
2. 顶部导航折叠
3. 标题区高度缩小

## 12. SEO 方案

### 12.1 首页 SEO

首页建议使用：

1. `title`：站点名称或自定义首页标题
2. `description`：站点简介
3. `keywords`：博客主关键词

### 12.2 文章详情 SEO

文章详情页建议直接使用文章表中的：

1. `seo_title`
2. `seo_description`
3. `seo_keywords`

若对应字段为空，则降级使用：

1. `title`
2. `description`
3. 标签或分类名拼接

## 13. 数据映射规则

### 13.1 首页文章卡片映射

建议映射如下：

- 标题：`info_article.title`
- 摘要：`info_article.description`
- 正文预览：`info_article.content`
- 发布时间：`info_article.ctime`
- 分类：`info_menu.name`
- 标签：`info_article.tag`

### 13.2 首页推荐映射

- 标题：`info_article.title`
- 链接：`/article/{id}`
- 过滤条件：`recommend_type=1`

### 13.3 详情页推荐映射

- 标题：`info_article.title`
- 链接：`/article/{id}`
- 过滤条件：同分类优先，推荐文章优先

## 14. 关键业务规则

### 14.1 导航层级

1. 当前阶段前台导航最多支持两级。
2. 顶级分类显示在导航栏。
3. 子分类显示在顶级分类下拉项中。

### 14.2 文章排序

1. 首页文章列表按 `ctime DESC`
2. 推荐文章列表默认也按 `ctime DESC`

### 14.3 正文输出

1. 首页不展示全文。
2. 详情页展示全文。
3. 正文若为 HTML，详情页需按 HTML 安全输出策略渲染。

### 14.4 安全输出

若后台允许文章内容保存 HTML，前台需特别注意：

1. 默认不直接信任用户输入 HTML
2. 若确认只有管理员写内容，可在受控前提下渲染 HTML
3. 更稳妥的方案是对 HTML 做白名单清洗后渲染

## 15. 推荐开发顺序

建议按以下顺序推进前台开发：

1. 前台路由与模板骨架
2. 顶部导航树
3. 首页文章列表与全站推荐
4. 文章详情页与点击计数
5. 点赞/点踩接口与前端交互
6. 分类页或分类聚合能力
7. SEO 和响应式细节优化

## 16. 本阶段交付建议

前台一期建议交付内容：

1. 首页 SSR 页面
2. 文章详情页 SSR 页面
3. 顶部两级导航
4. 首页全站推荐侧栏
5. 文章详情页类目推荐侧栏
6. 点击计数
7. 点赞与点踩
8. 基础 SEO

## 17. 总结

LaneBlog 前台最适合采用“Go + SSR + 轻交互”的实现方式。

该方案具备以下优点：

1. SEO 友好
2. 实现成本低
3. 与现有 Go 后端一致
4. 非常适合个人博客内容型站点

按此方案推进，可以在较短时间内完成一个可上线的前台版本。
